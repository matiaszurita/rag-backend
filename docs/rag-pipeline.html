01. Resumen

Que es ContextVault y que problema resuelve un RAG

ContextVault es un backend FastAPI para una aplicacion RAG. Permite autenticar usuarios, gestionar workspaces, subir documentos, indexarlos y consultarlos con retrieval + generacion controlada sobre contenido privado.

Problema

Un LLM general no conoce automaticamente los documentos privados de una aplicacion. Si responde solo con su conocimiento previo, puede omitir, inventar o descontextualizar informacion clave del usuario.

Solucion RAG

El backend primero recupera fragmentos relevantes del corpus del workspace y luego entrega ese contexto al modelo para generar una respuesta fundamentada. El resultado no es solo una respuesta: tambien incluye evidencia y metadata.

Idea central: ContextVault separa claramente dos momentos del pipeline: preparar documentos para busqueda y usar esos documentos como contexto para responder preguntas.

03. Upload de documentos

Archivo real en storage, metadata y estado en base de datos

Storage local

El contenido real del archivo se guarda en storage local bajo LOCAL_STORAGE_PATH. Este paso no genera embeddings ni chunks.

Metadata

La base de datos conserva workspace, nombre original, ruta de storage, content type, timestamps y estado del documento.

Separacion funcional

Upload rapido primero, indexacion despues. Esto evita perder el archivo si la indexacion falla y deja margen para futuras ejecuciones asincronas.

Estado Significado
uploaded El archivo existe, pero todavia no fue indexado.
indexing El backend esta extrayendo texto, creando chunks o generando embeddings.
indexed La indexacion termino correctamente y hay chunks persistidos.
index_failed Hubo un error durante la indexacion.
04. Indexacion

IndexDocumentService convierte archivos en chunks buscables

Pipeline interno de indexacion
  1. Validar ownership y workspace del documento.
  2. Marcar estado como indexing.
  3. Leer el archivo desde storage local.
  4. Extraer texto segun el tipo de archivo.
  5. Dividir el texto en chunks.
  6. Generar embeddings para cada chunk.
  7. Reemplazar chunks previos del documento, si existian.
  8. Persistir nuevos registros en document_chunks.
  9. Marcar el documento como indexed.

Extraccion de texto

El pipeline actual soporta archivos .txt, .md y .pdf. Primero se transforma el binario en texto utilizable.

Chunking

El splitter actual usa RecursiveCharacterTextSplitter, ajustado mediante RAG_CHUNK_SIZE y RAG_CHUNK_OVERLAP.

05. Embeddings y pgvector

Representacion semantica del texto para busqueda

Que es un embedding

Un embedding es una representacion numerica del significado aproximado de un texto. Permite comparar cercania semantica incluso cuando dos frases no usan exactamente las mismas palabras. 

Pregunta: "como configuro la clave secreta del JWT?"
Chunk:    "JWT_SECRET_KEY configures signing secrets"

Por que pgvector importa

ContextVault guarda embeddings en PostgreSQL y usa pgvector para ordenarlos por cercania vectorial. Eso habilita busqueda semantica dentro del workspace autenticado del usuario.

  • workspace_id
  • document_id
  • content
  • chunk_index
  • embedding
  • metadata
06. Retrieval

Tres modos de recuperacion: vector, keyword e hybrid

La orquestacion de retrieval pertenece a RetrievalService. El mismo comportamiento se comparte entre /rag/search y /rag/query.

Modo Como funciona Fortaleza principal Debilidad principal
vector Genera embedding de la pregunta y busca cercania semantica con pgvector. Encuentra intencion y sinonimos. Puede no priorizar terminos tecnicos exactos.
keyword Usa full-text search sobre document_chunks.content. Encuentra identificadores y literales exactos. No entiende reformulaciones como vector search.
hybrid Combina candidatos vectoriales y keyword con weighted RRF. Balancea significado e identificadores tecnicos. La interpretacion del score final es mas compleja.

Busqueda vectorial

Ideal para preguntas naturales o cuando el usuario consulta con palabras distintas a las del documento, pero con la misma intencion semantica.

Busqueda keyword / full-text

Ideal para terminos exactos como pgvector, JWT_SECRET_KEY, RAG_MIN_RELEVANCE_SCORE, workspace_id o document_chunks.

RAG hibrido

Reune ambos mundos: semantica para preguntas abiertas y precision lexical para identificadores, nombres de tablas, settings y clases tecnicas.

07. Weighted Reciprocal Rank Fusion

La fusion compara rankings, no scores crudos incompatibles

En hybrid retrieval, ContextVault no suma directamente el score de pgvector con el de full-text search. En su lugar, fusiona posiciones de ranking mediante weighted Reciprocal Rank Fusion, o weighted RRF.

Formula conceptual

score = vector_weight  * 1 / (RRF_K + vector_rank)
      + keyword_weight * 1 / (RRF_K + keyword_rank)

Si un chunk aparece alto en ambas listas, sube en el ranking final. Si aparece solo en una, igual puede competir con informacion parcial.

Variables relevantes

  • RAG_VECTOR_WEIGHT
  • RAG_KEYWORD_WEIGHT
  • RAG_VECTOR_CANDIDATES
  • RAG_KEYWORD_CANDIDATES
  • RAG_RETRIEVAL_MODE
Aclaracion importante: RRF_K = 60 es la constante interna actual del servicio de retrieval. Hoy no esta expuesta como variable de entorno y no existe RAG_RRF_K en esta fase.
08. Reranking

Capa preparada en arquitectura, pero sin adapter real todavia

pregunta
   -> retrieval obtiene candidatos iniciales
   -> reranker puede reordenar candidatos
   -> se seleccionan los mejores chunks
   -> se construye contexto
   -> LLM genera respuesta

Donde entra

El reranking ocurre despues del retrieval y antes de construir el contexto que se envia al LLM. Su rol es reordenar candidatos, no reemplazar el retrieval.

Estado actual del proyecto

El backend ya define RerankerPort y puede componer un provider por dependencias. Actualmente el provider por defecto es NoOpReranker, que conserva el orden existente.

Importante: la arquitectura ya esta lista para un adapter real de reranking en el futuro, pero el pipeline actual no usa todavia un modelo real de reranking semantico. Esa extension sigue siendo una fase posterior.
09. /rag/search vs /rag/query

Debug de retrieval vs pregunta-respuesta con LLM

/rag/search

Endpoint de debug y retrieval. Devuelve chunks recuperados, score breakdowns, modo de retrieval y metadata para inspeccionar el comportamiento del sistema. No genera respuesta con LLM.

/rag/query

Endpoint de pregunta-respuesta. Usa el mismo retrieval, selecciona contexto, construye el prompt con RagPromptBuilder, llama a Gemini y devuelve answer, sources y metadata.

Flujo de /rag/query
  1. Validar pregunta y workspace.
  2. Recuperar chunks con RetrievalService.
  3. Si el modo es vector, aplicar RAG_MIN_RELEVANCE_SCORE.
  4. Si el modo es keyword o hybrid, no usar ese threshold vectorial.
  5. Si no hay contexto suficiente, devolver respuesta controlada sin llamar al LLM.
  6. Construir prompt con el contexto recuperado.
  7. Llamar a Gemini mediante LLMProviderPort.
  8. Devolver respuesta, sources y metadata.
10. Sources y metadata

Evidencia, trazabilidad y diagnostico del retrieval

En un sistema RAG no alcanza con responder bien: tambien importa explicar de donde salio la evidencia y con que pipeline fue seleccionada.

score

Score final de ordenamiento.

vector_score

Score original de busqueda vectorial.

keyword_score

Score original de busqueda keyword.

rerank_score

Score opcional asignado por el reranker.

original_rank

Posicion antes del reranking.

reranked_rank

Posicion despues del reranking.

retrieval_source

Indica vector, keyword o hybrid.

fusion_algorithm

Hoy, weighted_rrf cuando aplica.

context_chunks_used

Cantidad de chunks usados en el prompt final.

llm_model

Modelo LLM usado para generar la respuesta.

reranking_enabled

Si el request o config pidio reranking.

reranking_provider

Provider configurado; hoy, noop.

reranking_applied

Indica si efectivamente se aplico el paso.

Diferencia util: en /rag/search se devuelve el contenido completo del chunk para debug. En /rag/query las sources muestran un content_preview para evidenciar sin exponer todo el contexto usado.

12. Que no hace todavia

Limites actuales del pipeline

No hay adapter real de reranking

La extension ya tiene port y contrato, pero el provider efectivo por defecto sigue siendo NoOpReranker.

No hay chat conversacional

El sistema actual responde preguntas puntuales. No mantiene historial de conversacion como parte del pipeline RAG vigente.

No hay parent-child chunks

El chunking actual es lineal. Todavia no existe una estrategia padre-hijo para recuperar contexto jerarquico.

No hay streaming

La respuesta del LLM se devuelve como payload completo, no como stream de tokens o fragmentos incrementales.

No hay observabilidad avanzada

Existen sources y metadata utiles, pero no una capa completa de tracing, dashboards operativos o analitica avanzada de retrieval.

13. Proximas fases

Extensiones naturales del roadmap RAG

Fase 6: chat conversacional

Incorporar historial y continuidad entre preguntas para construir contexto a lo largo de una sesion conversacional.

Fase 7: parent-child chunks

Recuperar fragmentos pequenos manteniendo la posibilidad de expandirlos a una unidad de contexto mayor y mas coherente.

Fase futura: adapter real de reranking

Reemplazar el comportamiento noop por un adapter real detras de RerankerPort, sin mover la logica fuera de la arquitectura actual.

15. Ejemplos curl

Pruebas manuales de retrieval y query answering

curl -X POST "http://localhost:8000/api/v1/workspaces/$WORKSPACE_ID/rag/search" \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "JWT_SECRET_KEY",
    "top_k": 5,
    "retrieval_mode": "hybrid",
    "reranking_enabled": true
  }'
curl -X POST "http://localhost:8000/api/v1/workspaces/$WORKSPACE_ID/rag/query" \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "question": "Para que sirve JWT_SECRET_KEY?",
    "top_k": 5,
    "retrieval_mode": "hybrid",
    "reranking_enabled": true
  }'
Variables del ejemplo: ACCESS_TOKEN es el JWT obtenido desde /api/v1/auth/login y WORKSPACE_ID es el identificador del workspace que contiene los documentos indexados.
